# 인플링 (Inflring) 코딩 규칙

## 1. 언어 원칙
- 모든 문서, 주석, 커밋 메시지, UI 텍스트는 한국어로 작성한다.
- 코드 식별자(변수, 함수, 클래스, 파일명)는 영어로 작성한다.
- 변수/함수: camelCase, 컴포넌트/클래스: PascalCase.

## 2. 코드 작성
- TypeScript strict 모드 사용. `any` 타입 사용 금지.
- `console.log` 프로덕션 코드 금지 (디버깅 후 반드시 제거).
- 함수는 50줄 이하, 파일은 300줄 이하. 초과 시 분리.
- 단일 책임 원칙: 함수/컴포넌트 하나가 하나의 역할만 수행.
- 네이밍: 컴포넌트 `PascalCase`, 유틸 `camelCase`, 상수 `UPPER_SNAKE_CASE`, API 라우트 `kebab-case`.
- ESLint + Prettier 설정 준수. 커밋 전 린트 통과 필수.

## 3. 아키텍처
- 3계층 구조: `app/api/` (HTTP 핸들링) → `lib/services/` (비즈니스 로직) → `lib/repositories/` (Supabase 접근).
- 역방향 의존 금지: 리포지토리가 서비스를, 서비스가 API를 참조하면 안 된다.
- API 응답 통일: `{ success: boolean, data?: T, error?: string, meta?: object }`.
- RESTful 원칙 준수: GET(조회), POST(생성), PATCH(수정), DELETE(삭제).
- Supabase 클라이언트는 `lib/supabase/` 에서만 생성하고 재사용한다.

## 4. 최소 구현
- 요청받은 기능만 구현한다. 임의 UI 개선, 추가 API, 부가 기능 금지.
- 변경 범위를 최소화한다. 관련 없는 파일 수정 금지.
- MVP 우선: 동작하는 최소 기능 → 검증 → 확장.

## 5. 트러블슈팅
- 근본 원인을 찾아 해결한다. 증상만 가리는 우회 금지.
- 금지 패턴: try-catch로 에러 삼키기, `as any` 타입 변환, 기능 비활성화로 에러 회피.
- 해결된 트러블슈팅 교훈은 이 파일 9번 섹션에 누적 기록한다.
- 에러 발생 시: 로그 확인 → 재현 → 원인 파악 → 수정 → 테스트 순서.

## 6. 토큰 효율
- 불필요한 .md 파일 남발 금지. 문서는 필요한 것만 생성.
- 주석은 "왜(Why)" 이유만 기록. "무엇(What)"은 코드가 설명.
- 명확한 네이밍으로 주석 필요성을 줄인다.
- 중복 코드보다 적절한 추상화, 단 3곳 미만이면 중복 허용.

## 7. 과도한 일반화 금지
- YAGNI 원칙: 현재 필요한 것만 구현. 미래 요구사항 추측 금지.
- 추상화는 동일 패턴이 3곳 이상에서 사용될 때만 적용.
- 설정 파일, 팩토리 패턴 등 과잉 엔지니어링 지양.
- 간단한 코드가 복잡한 추상화보다 낫다.

## 8. 버전 관리
- 커밋 메시지 형식: `[타입] 한국어 설명` (예: `[feat] 로그인 페이지 구현`).
- 타입: feat, fix, refactor, docs, style, test, chore.
- 하나의 커밋 = 하나의 논리적 변경. 거대 커밋 금지.
- main 브랜치 직접 작업 금지. feature 브랜치 사용.
